# AGENTS

# Purpose

This project is a modern, interactive 3D portfolio built with Next.js, TypeScript, React Three Fiber, and Tailwind CSS.

Your goal is to help build a production-quality application that is performant, maintainable, and easy to evolve.

Prioritize:

* Correctness
* Maintainability
* Performance
* Readability
* Reusability
* Accessibility
* Simplicity

---

# Workflow

Before making changes:

1. Understand the existing implementation.
2. Analyze the relevant files before editing.
3. Reuse existing components, hooks, and utilities whenever possible.
4. Preserve existing architecture and conventions.
5. Make the smallest change that solves the problem.

Avoid unnecessary rewrites.

---

# Implementation Guidelines

When implementing features:

* Keep components focused on a single responsibility.
* Prefer composition over duplication.
* Keep logic predictable and easy to follow.
* Avoid introducing unnecessary abstractions.
* Follow existing project patterns before creating new ones.
* Avoid modifying unrelated code.

---

# Decision Making

When multiple solutions exist:

Prefer the solution that is:

1. Consistent with the current codebase.
2. Easier to maintain.
3. Easier to understand.
4. Better for long-term scalability.
5. Better for runtime performance.

Do not introduce new libraries or architectural patterns unless they provide a clear benefit.

---

# Problem Solving

When debugging:

* Identify the root cause before making changes.
* Avoid speculative fixes.
* Explain assumptions when they cannot be verified.
* Preserve existing behavior unless intentionally changing it.

---

# Code Quality

Generated code should:

* Be production-ready.
* Be easy to read.
* Avoid unnecessary complexity.
* Avoid duplicate logic.
* Respect the existing architecture.
* Keep public APIs consistent whenever practical.

---

# Safety

Never:

* Hardcode secrets or credentials.
* Invent APIs or project files.
* Remove security checks without justification.
* Ignore errors silently.

If required information is missing, ask for clarification instead of guessing.

---

# Documentation

When project structure or public behavior changes:

* Update the relevant documentation.
* Keep documentation concise and synchronized with the implementation.

Do not update documentation for purely internal refactors unless it becomes inaccurate.

---

# Response Format

When completing a task:

* Summarize what changed.
* List important modified files.
* Mention assumptions made.
* Mention notable performance implications, if any.
* Mention follow-up work only if it adds meaningful value.

---

# When Unsure

If the requested behavior is ambiguous:

* State the uncertainty clearly.
* Explain the available options.
* Request clarification before making assumptions.
